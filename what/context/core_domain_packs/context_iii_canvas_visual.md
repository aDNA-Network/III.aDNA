# III Domain Pack: Canvas Visual (DRAFT)

Domain-specific trap library for reviewing canvas-based visual deliverables generated by `PresentationBuilder` and `ComicPageBuilder`. Covers presentation decks, comic pages, and any future canvas-as-communication output. Extends the `core` and `web_design` packs with canvas-specific failure modes derived from VR1-VR5 rubric, M00 walkthrough findings (Herb + Carly Moore, 2026-04-10), and the canonical 8 traps enumerated in `lattice-labs/how/campaigns/campaign_canvas_visual_command/CLAUDE.md`.

**Status: draft, campaign-scoped.** Graduation to canonical status requires each trap firing in ≥3 cycles with ≥80% user acceptance per the III graduation criteria.

## When to Load

Load alongside `core` (and optionally `web_design`) when reviewing:

- Presentation decks generated by `PresentationBuilder` (any theme)
- Comic pages generated by `ComicPageBuilder` (Issue #1 pilot, future issues)
- HTML renderings produced by `canvas_html_renderer.py`
- Playwright screenshots of any canvas surface
- Multi-canvas adversarial review sessions (Phase 2 RLHF)
- Comic review canvases (`pilot_review.canvas` and successors)
- Any III review where the deliverable artifact is `.canvas`, rendered HTML of a canvas, or a screenshot of either

## Canonical Reference

Visual scoring rubric is the **VR1-VR5 weights** maintained in `lattice-protocol/tests/fixtures/baseline_vr_scores.json`:

| Dimension | Weight | What it measures |
|---|---|---|
| **VR1 Text Readability** | 25% | font size, contrast, line length |
| **VR2 Visual Hierarchy** | 25% | heading prominence, information flow |
| **VR3 Whitespace Quality** | 20% | breathing room, spacing rhythm |
| **VR4 Color Harmony** | 15% | cohesive palette, accent balance |
| **VR5 Professional Polish** | 15% | alignment, sizing, enterprise-ready |

Pass threshold: **7.5 weighted**. Current synthetic baseline: 105 cells (7 themes × 15 slide types) all in 7.5–8.5 range. Wilhelm deck baseline: TBD (M01 deliverable).

Walkthrough findings reference: `how/campaigns/campaign_canvas_visual_command/missions/mission_cvc_m00.md` (F1-F11).

## Canvas Visual Traps

### Trap CV-COHERENCE-01: Visual Consistency Drift

| Field | Value |
|-------|-------|
| **Description** | Color, font, spacing, or layout rhythm drifts between slides/panels in a deliverable that should read as one coherent piece. The drift may be intra-deck (slide 3 looks like it came from a different deck than slide 7) or intra-page (panel 4 uses a different palette than panel 1). |
| **Example** | Wilhelm deck slides 1-3 use identical heading-margin rhythm (96px), then slide 4 uses 64px without an obvious reason. Or comic page 5 panel 2 uses warm Stanley palette and panel 3 shifts to cool palette mid-beat. |
| **Where It Hides** | Per-slide style overrides that bypass `canvas_design_tokens`, hand-edited `styleAttributes` on a single canvas node, theme tokens half-applied to a comic builder, palette drift during a multi-call image generation pass. |
| **Severity Default** | Medium |

**Detection method**: Render all slides/pages to HTML via `canvas_html_renderer`, screenshot via Playwright, then prompt Gemini to compare slides 1↔N for: (a) heading style consistency, (b) body type consistency, (c) margin/padding rhythm, (d) color palette match. Any flagged drift between slides triggers the trap. For comics, run the same comparison panel-to-panel within a page and page-to-page within a chapter.

**Why it happens**: The visual loop hasn't been wired (Phase 0 problem). Once `canvas_html_renderer` + Playwright + Gemini run on every deliverable, drift gets caught at the rendering stage instead of the publishing stage.

### Trap CV-AUDIENCE-01: Audience-Presentation Fit Ambiguity

| Field | Value |
|-------|-------|
| **Description** | A deck or comic page can't decide whether it's for a live audience (sparse, story-driven, ~20 words/slide) or async/scannable consumption (dense, self-contained, can be 100+ words/slide). The mixed signals make every slide feel slightly wrong for whichever audience the reviewer assumes. |
| **Example** | Wilhelm slides 1-3 read as keynote (sparse, story-driven) but slide 4 (Architecture) reads as async (dense, layered, three-tier stack with all layers crammed onto one slide) — Carly: _"depending on the audience, you might need like one slide for each of these. Some people will read this and get it right away. Some will be like 'I am completely lost.'"_ The deck doesn't commit. |
| **Where It Hides** | Mission briefs that don't specify audience context, decks generated from research dumps without an audience parameter, comic scripts that target multiple reader speeds without explicit chapter pacing. |
| **Severity Default** | High |

**Detection method**: Score every slide against an "average words per node" + "complexity per node" metric. Compute the variance across the deck. High variance → audience ambiguity. Cross-check by prompting Gemini: _"Is this slide intended to be read aloud (keynote) or read silently (async)?"_ Inconsistent answers across slides → trap fires. For comics: same approach but on panel density and beat-to-beat pacing.

**Why it happens**: Builders default to a single style without an `audience` parameter. PresentationBuilder needs `AudienceContext` to be threaded through scoring (it exists in `canvas_config.py` but isn't enforced).

### Trap CV-CONFIDENCE-01: Content Confidence Inflation

| Field | Value |
|-------|-------|
| **Description** | Slide or panel copy presents projective claims, future capabilities, or aspirational metrics as established fact. The inflation is usually in verb tense ("Lattice does X" vs "Lattice can do X") or in framing ("we have demonstrated" vs "we are validating"). |
| **Example** | "Lattice federates 50+ partner nodes" when the actual count is 2 prototype nodes + 4 planned. Or "Stanley diagnoses rare disease cases" when SS aDNA is mid-Phase-4 build, not in clinical use. Or comic dialogue: _"Stanley solves the cold-start problem"_ when the comic depicts the framing of the problem, not its solution. |
| **Where It Hides** | Headline slides, vision slides, marketing copy lifted into slide bodies, comic dialogue written for narrative effect rather than factual accuracy, footnote-free statistics, AI-generated summaries of in-progress work. |
| **Severity Default** | High |

**Detection method**: Scan all slide titles + body text + comic dialogue for verb-tense inflation (`will → does`, `plan to → has`, `validating → demonstrated`). Check claim numbers against a campaign ground-truth source (LFS state file, federation_state.json, or similar). For each metric on a slide, ask: _"Is this a current count, a target count, or an aspirational count? Is the framing honest about which?"_

**Why it happens**: Marketing voice creeps into operational decks. PRDs get retrofitted into slide form without re-verifying the claims. Comic narratives prefer dramatic certainty over honest uncertainty.

### Trap CV-HIERARCHY-01: Visual Hierarchy Inconsistency

| Field | Value |
|-------|-------|
| **Description** | H1 (slide title), H2 (section header), and body text collapse to similar visual weights, making it hard to find the title-equivalent on every slide. Or the hierarchy is consistent within a slide but inconsistent across slides — slide 2 has a clear title, slide 5 has the title buried inside a content block. |
| **Example** | Wilhelm slide 4 (Architecture) — title "Architecture: Open Composable Deployable" doesn't visually dominate the dense three-tier stack content. Reviewer can't tell at a glance whether the slide is about "Architecture" or "Open Composable Deployable" or one of the three layers. |
| **Where It Hides** | Slide builders that don't enforce H1 typography rules, slides where the title was edited to be too long (pushes content up into the title region), `slide-type-content` overriding the title style, custom slide types from `canvas_slide_builders.py` that bypass the design token system. |
| **Severity Default** | Medium |

**Detection method**: For each rendered slide, extract the largest text element by computed font-size. Verify it matches the canonical title slot for that slide type. Verify the H1 → H2 → body ratio is at least 2:1.5:1 (per design tokens). Cross-reference Carly's gestalt rule: _"can you find the title in 0.5 seconds?"_ If a Gemini scoring pass can't identify the slide title, the hierarchy has collapsed.

**Why it happens**: Slide-type-specific builders (`canvas_slide_builders.py`) have drifted from a single typography contract. Each builder rebuilds the title region from scratch instead of inheriting a `TitleNode` primitive.

### Trap CV-TEMPLATE-01: Template Voice / Generic Filler

| Field | Value |
|-------|-------|
| **Description** | Slide or panel copy that could apply to any AI/biotech/SaaS company without modification — generic filler that survived the entity-substitution test (replace "Lattice" with "Acme" and the slide still makes sense). Often appears in section dividers, transition slides, and CTA slides. |
| **Example** | "Our platform enables your success" (generic — substitute Acme, still works). "The future is here" (template). "Join us on the journey" (template). Contrast with: "Wilhelm Foundation's open-source rare disease diagnostic stack" (specific, doesn't substitute). |
| **Where It Hides** | Section dividers, "Join" slides, transition slides between content blocks, footer text on every slide, AI-generated summaries of long content, slide bodies generated from a 1-line prompt with no domain context. |
| **Severity Default** | Medium |

**Detection method**: Apply the entity-substitution test from `web_design` Trap 2 — replace "Lattice" / "Wilhelm" / "Stanley" / domain-specific entities with generic placeholders ("Acme", "Foundation", "Doctor"). If the slide reads naturally with the substitution, the copy is generic. Run on every slide automatically as part of the III dispatch.

**Why it happens**: Slide bodies are sometimes auto-generated from short prompts that the LLM defaults to filling with safe corporate language. The fix is to thread domain-specific context into the slide body generation, not to manually edit each slide.

### Trap CV-CONTRAST-01: WCAG Contrast Failures

| Field | Value |
|-------|-------|
| **Description** | Text/background combinations that fail WCAG AA contrast requirements (4.5:1 for body, 3:1 for large/display text, 3:1 for interactive controls). Affects accessibility, projector readability, colorblind viewers, and aged display panels. |
| **Example** | `science_stanley` theme accent yellow `#e9d97a` on `#1a1b26` body — borderline pass. Or muted body text `#9098b3` on `#24283b` background — fails AA at 3.2:1. Or comic speech bubble white text on a busy hero panel without a scrim. |
| **Where It Hides** | Theme palette accent colors used as body text, muted/disabled state colors, comic dialogue overlaid directly on photographic panels, footnote text, slide footers, decorative subtitle slots. |
| **Severity Default** | High |

**Detection method**: For every text element in a Playwright screenshot, compute the contrast ratio against the resolved background color (or average background pixel values when over an image). Flag <4.5:1 for body, <3:1 for display (font-size > 18pt). Run automatically via Playwright `getComputedStyle` + WCAG contrast formula. For images, sample the screenshot pixels under each text bounding box.

**Why it happens**: Themes are designed for the synthetic 105-cell baseline, not for content-heavy real decks where body text drifts onto unexpected backgrounds. Comic panels generate at variable tones — speech bubbles need automatic scrim/halo logic.

### Trap CV-PENDING-01: Unresolved Placeholders Shipped as Done

| Field | Value |
|-------|-------|
| **Description** | A `PendingImage` or `PendingPanel` placeholder survives the build pipeline and ships in the final deliverable. The slide claims to be done but has a "[image placeholder]" or empty image slot or unrendered prompt. |
| **Example** | Wilhelm deck currently has Visual Ratio 0% — every image slot is unfilled (M02 will wire Imagen 4). Until then, every slide that "should" have an image is implicitly pending. After M02: any slide where `PendingImage.resolved == False` at render time should fire this trap. |
| **Where It Hides** | Slide nodes with `pending_image_id` set but no resolved URL, comic panels with `PendingPanel` markers in the canvas JSON, slides where the prompt was generated but Imagen 4 hasn't been called, deliverables marked "ready for review" in a campaign doc but pre-Imagen wiring. |
| **Severity Default** | High (Critical if shipped to customer) |

**Detection method**: Scan canvas JSON for `PendingImage` / `PendingPanel` markers before declaring any slide "done". After M02: scan rendered HTML for `[pending]` / `data-pending="true"` attributes. Cross-reference build report's `unresolved_pending_count` field — if non-zero, every dependent slide fires this trap.

**Why it happens**: The image lifecycle hasn't been wired (M02 is the fix). Until then, "done" means "structurally done" not "visually done", and the gap is invisible without explicit checks.

### Trap CV-COMIC-STYLE-01: Comic Style Lock Drift

| Field | Value |
|-------|-------|
| **Description** | Character appearance, palette, line weight, or rendering style drifts between panels, pages, or chapters of a comic. Most commonly: a character looks recognizably like themselves on page 1 and recognizably different on page 4. Or the color script shifts mid-scene without a deliberate beat. |
| **Example** | Science Stanley pilot first panel uses a different art style than panels 2-N (M00 finding from Carly: _"aside from maybe the first panel, it looks like a different art style too"_). Or a hero panel's palette is warm sunset, then the very next panel cuts to cool blue without a transition beat. Or Stanley's lab coat has a different cut on page 5 than page 2. |
| **Where It Hides** | Multi-call image generation passes where the seed/prompt drift between calls, panels regenerated individually after a global pass, character reference sheets that aren't enforced as constraints, color script files that exist but aren't injected into the per-panel prompt. |
| **Severity Default** | High |

**Detection method**: For each comic page rendered, extract per-panel screenshots and prompt Gemini: _"Does the character {NAME} on panel N look the same as panel N-1? Is the color palette consistent between adjacent panels? Is the line weight and rendering style consistent?"_ For multi-page comparisons: same prompt across page boundaries. For style lock specifically, also run a perceptual diff (`canvas_print.py` may already have this; if not, build it from a Playwright screenshot + ImageHash).

**Why it happens**: `ComicPageBuilder` 6-layer prompts don't enforce a character constraint at the per-panel level. Each panel gets a fresh generation with reference text, but reference text alone isn't enough to lock style. M07 (Comic Visual Review Extension) is the systematic fix.

### Trap CV-INSIDER-CONTEXT-01: Insider-Context Contamination (NEW from M00)

| Field | Value |
|-------|-------|
| **Description** | A reviewer brings prior context to a deliverable and fills in gaps the fresh viewer cannot. The reviewer reports the deliverable as "clear" while a stranger sees it as "broken" or "incomplete". The trap fires when the reviewer's confidence in the deliverable's quality is grounded in context the deliverable doesn't carry. |
| **Example** | M00 walkthrough — both Herb and Carly read the comic pilot canvas as making sense because they had worked on Issue #1 prior. Carly: _"I am able to fill in some context because we worked on it before."_ Herb caught himself doing the same thing. The canvas itself doesn't communicate the story to a new viewer; the existing reviewers were silently filling in. **High-confidence walkthrough finding** — both reviewers converged independently. |
| **Where It Hides** | Internal review sessions where reviewers know the campaign history, decks for partner meetings where the reviewer assumes the partner has read the briefing, comic review canvases that mirror existing pages without adding editorial context, "ready for board" decks reviewed only by team members. |
| **Severity Default** | High (Critical when the deliverable is for external stakeholders) |

**Detection method**: For every III review of a canvas deliverable, the reviewer must explicitly answer: _"What context am I bringing that a fresh viewer would NOT have?"_ Then re-score the deliverable as if that context were absent. If the score drops by ≥1.0 weighted, the trap fires. Alternative automated detection: prompt Gemini with the canvas + an explicit "fresh viewer with no prior context" framing, compare to the reviewer's score, flag deltas > 1.0.

**Why it happens**: Insider review is the default review mode in a small team. The fix is procedural: every review session needs a fresh-viewer framing pass before the reviewer's natural read.

**M00 evidence**: Both Herb and Carly converged on this independently while reviewing `pilot_review.canvas`. Routed F11 (selective panel curation > full-page duplication) to `campaign_comic_production_excellence` partly because of this pattern.

### Trap CV-DIMENSION-VISIBILITY-01: Slide Dimensions Hidden (NEW from M00 / F7)

| Field | Value |
|-------|-------|
| **Description** | The canvas builder produces slides without exposing slide-dimension metadata or visible dimension borders. Reviewers cannot evaluate "is this too much text" against any standard because they cannot see what the slide's dimensions are relative to (PowerPoint 16:9, 4:3, custom). The "too much text" judgment becomes impossible without a frame of reference. |
| **Example** | M00 walkthrough — Herb: _"I am not sure if these size of these boxes is relative to like a dimension like, like the size of a Windows PowerPoint or is this a specific aspect ratio. And if it isn't, then I think it's hard to know like the spacing of a page. Like how much words is too much for a page."_ The slide boxes don't read as PowerPoint-dimensioned, and inside-box sizes vary across slides. F7 in M00 findings table. |
| **Where It Hides** | `CanvasBuilder` output that uses Obsidian canvas coordinates without an explicit aspect-ratio frame, `slide-type-content` builders that don't enforce a 16:9 viewport, HTML rendering that lets slides flow at arbitrary widths, screenshot output cropped to content rather than to a slide-dimension bounding box. |
| **Severity Default** | Medium |

**Detection method**: For each slide in a canvas, verify the bounding box matches a known aspect ratio (16:9, 4:3, A4 landscape, US Letter landscape). For HTML renderings, verify the rendered slide is wrapped in a viewport element with explicit dimensions. For screenshots, verify each slide image has explicit pixel dimensions matching a known projection format. If any slide fails, the trap fires for that slide.

**Why it happens**: Obsidian canvas is a free-form spatial workspace. The builders inherit that freedom and produce slides without enforcing a presentation aspect ratio. The fix (M01 or M03 deliverable): add a `slide_frame` indicator that's visible to reviewers — either a 16:9 bounding box drawn around the slide content area or explicit dimension metadata in the rendered HTML.

**M00 evidence**: F7 in the M00 findings table. Logged from Herb's walkthrough verbatim quote. Carly converged: _"if it's this whole box, then that's not very much text. But if it's confined to that space that it takes up within the box, then it's pretty full."_

## Cross-Trap Escalation Rules

When multiple traps fire on the same location:

| Combination | Escalation |
|---|---|
| CV-COHERENCE-01 + CV-HIERARCHY-01 | → High (visual system has multiple inconsistencies — likely a refactor regression) |
| CV-AUDIENCE-01 + CV-TEMPLATE-01 | → High (deck has neither audience commitment nor specific voice — likely auto-generated boilerplate) |
| CV-CONFIDENCE-01 + CV-PENDING-01 | → Critical (deck claims completion that isn't actual — both content and visual fronts) |
| CV-COMIC-STYLE-01 + CV-CONTRAST-01 | → Critical (comic readability + accessibility both broken) |
| CV-INSIDER-CONTEXT-01 + any external-stakeholder context | → Critical (deliverable will fail for the audience it's actually for) |
| CV-DIMENSION-VISIBILITY-01 + CV-HIERARCHY-01 | → High (reviewers literally cannot evaluate the slide because the frame and the title are both broken) |

## Trap Graduation

This pack starts as **draft** with 10 initial traps (8 canonical from campaign CLAUDE.md + 2 from M00 walkthrough). Graduation criteria (per III protocol):

1. Each trap must fire in ≥3 cycles with user acceptance ≥80%
2. The trap pack as a whole must surface a finding the `core` + `web_design` packs missed
3. Trap detection must be reproducible across multiple surface types (presentation deck + comic page + canvas-as-communication diagram)

Traps that graduate become part of a permanent `canvas_visual` pack (or get absorbed into `core` if they generalize beyond canvas).

Traps that don't graduate after 10 cycles are either revised or retired.
